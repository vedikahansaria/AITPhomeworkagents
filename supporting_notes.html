Implementation Logic & Methodology

These notes document the technical decisions made during the creation of the Churn Prediction model.

1. Data Cleaning Strategy

Raw data is rarely ready for modeling. The script handles missing values by imputing the median (for numerical data) or the mode (for categorical data). This prevents the model from crashing due to null values.

2. Feature Engineering

Machine Learning models require numerical input. We applied One-Hot Encoding to categorical variables (like turning "Yes/No" into 1/0). This allows the algorithm to mathematically weigh categorical differences.

Why Scaling? We used StandardScaler on numerical features (like Tenure or Monthly Charges) to ensure that features with larger ranges don't dominate the objective function of the algorithm.

3. Model Selection: Logistic Regression

We selected Logistic Regression for the baseline model because:

  • It is highly interpretable (we can see which features impact churn most).
  • It is computationally efficient.
  • It provides probability scores, not just binary classifications.

4. Evaluation Metrics

Accuracy alone is misleading in churn prediction because usually, most customers don't churn. Therefore, we focus on:

  • Recall: Of all the people who actually churned, how many did we correctly identify? (Crucial for retention).
  • Precision: Of the people we labeled as "At Risk," how many actually churned?
